<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| legacy `templates/` / `static/` → copied adminui tree | Phase 6 要复制现有前端，但不能污染 source 文件 |
| copied asset tree → later Go runtime wiring | 如果 contract 不清晰，06-02 以后会在 mount path 和 asset roots 上漂移 |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01-01 | T | legacy source files | mitigate | 只在 `backend-go/internal/adminui/` 创建 copied assets，并用 manifest 记录 source/target/hash；不修改 `templates/`、`static/`、`src/web/app.py`。 |
| T-06-01-02 | T | copied asset inventory | mitigate | route/manifest schema tests 只允许当前 Phase 6 范围内的模板和静态资源进入 copied tree，防止偷偷带入额外文件。 |
| T-06-01-03 | R | frontend copy provenance | mitigate | `copy_manifest.json` 保留每个 copied file 的来源和 hash，后续 Phase 7-9 改动可以基于基线追溯而不是凭记忆操作。 |
</threat_model>

<risk_and_rollback>
## 风险

- copied tree 范围定义不完整，会让 06-02 以后出现“页面能渲染但缺 JS/CSS”的假启动状态。
- 未建立 manifest 追踪的话，后续 shell 改造会掩盖“复制基线”与“重构改动”的边界。

## 回滚点

- 回滚只删除 `backend-go/internal/adminui/` 新增树，不回滚或覆盖 legacy `templates/`、`static/`。
- 如果复制范围判断错误，修正 manifest 和 copied tree 后重跑 `internal/adminui` 测试；不允许通过修改 legacy source 来“补齐”。
</risk_and_rollback>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser/static-js→payment service | Existing payment and accounts pages send account IDs, bind-card task actions, and session/subscription writes into the Go payment slice. |
| payment transition adapters→external payment/browser/session runtime | Transition adapters may talk to browser automation, session bootstrap logic, or temporary Python-backed helpers. |
| payment service→accounts persistence | Payment flow writes session token, cookies, and subscription state into the shared accounts truth source. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04-01 | T | payment transition adapters | mitigate | Keep every external/runtime call behind `internal/payment/transition_adapters.go`, normalize errors there, and add tests that fail on missing-seam regressions. |
| T-04-04-02 | I | session/subscription writeback | mitigate | Reuse the shared accounts repository and preserve current masked/detail response behavior so session tokens, cookies, and card secrets are not echoed to normal page loads. |
| T-04-04-03 | D | browser/session/bootstrap runtime | mitigate | Bootstrap tests must exercise helper-built services and fail fast on omitted adapters, preventing mounted routes from degrading into runtime-configuration dead ends. |
</threat_model>

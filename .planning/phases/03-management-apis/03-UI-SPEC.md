# Phase 3 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> Phase 3 keeps the existing Jinja templates, shared CSS tokens, and page-scoped JavaScript intact. No frontend rewrite, no SPA conversion, and no component-library introduction in this phase. Source: `REQUIREMENTS.md` (`CUT-01`), `ROADMAP.md` (Phase 3), `03-CONTEXT.md` (`D-02`).

## Layout And Component Inventory

- Shared shell is fixed: `site_notice` -> sticky top navbar -> `page-header` -> stacked cards/panels inside `.container` with `max-width: 1280px`.
- Existing navigation order is fixed: `注册` / `账号管理` / `账号总览` / `邮箱服务` / `支付` / `卡池` / `自动进team` / `后台日志` / `设置` / `退出`.
- Reusable management primitives are fixed: `stats-grid`, `card`, `card-header`, `card-body`, `toolbar`, `data-table`, `pagination`, `tabs`, `modal`, `dropdown`, `toast`, `empty-state`, `skeleton`, `info-grid`.
- Current page-local exceptions stay local:
  - `/accounts-overview` keeps its blue-violet card-management module and icon-toolbar language.
  - `/logs` keeps its mono console panel inside the normal shell.
  - Dense account status dots in `/accounts` remain allowed in table cells only.
- Existing payment/team entry points remain visible where they already exist. Phase 3 preserves placement and affordance only; downstream flow ownership stays Phase 4.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gaps, badge padding offsets, micro spacing |
| sm | 8px | Compact control spacing, dropdown item gaps, toolbar micro groups |
| md | 16px | Default control spacing, form groups, table cell padding |
| lg | 24px | Card padding, section gaps, toolbar separation |
| xl | 32px | Page header to content spacing, major internal blocks |
| 2xl | 48px | Reserved for major panel separation only; use for new large management subsections if needed |
| 3xl | 64px | Reserved for page-level breathing room only; do not retrofit existing pages without a parity reason |

Exceptions: the reusable spacing scale is limited to the tokens above. Existing shipped dense controls that already use legacy component-specific heights in icon-only card actions or `accounts_overview` search/select controls remain frozen parity exceptions, not reusable spacing tokens. Preserve them only in place; do not introduce additional custom heights.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f8fafc` | Page background, canvas, empty surrounding surface |
| Secondary (30%) | `#ffffff` | Cards, navbar, modal panels, table surfaces |
| Accent (10%) | `#10a37f` | Primary buttons, active nav/tab states, focus rings, selected/active highlights, success-forward emphasis |
| Destructive | `#ef4444` | Delete, clear, hard-failure and irreversible actions only |

Accent reserved for: primary submit buttons, active navigation/tab states, focus outlines, selected-card/list emphasis, progress and success confirmation accents. Do not use accent green for neutral secondary buttons, table row hover, warning/info stats, or destructive actions.

Preserved local exception: `/accounts-overview` may keep its existing blue-violet gradient cluster (`#6d28d9` -> `#8b5cf6`) and cool blue panel surfaces because that language is already shipped there. Do not spread that palette to `/accounts`, `/settings`, `/email-services`, or `/logs`.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `保存设置` for persistent configuration forms; modal edit/create submits must use action+object labels such as `保存代理配置`, `添加邮箱服务`, `导入账号数据`, and `添加账号` |
| Empty state heading | `暂无数据` |
| Empty state body | `没有找到符合条件的记录。请调整筛选条件，或先添加配置/导入数据。` |
| Error state | Page/block loading uses `加载失败，请检查网络连接后重试。`; action toasts use `失败: {具体原因}` |
| Destructive confirmation | `删除账号/服务/代理`: `确定要删除 ... 吗？此操作不可恢复。`; `导入数据库`: `系统会先自动备份当前数据库，再执行覆盖导入。`; `清理/清空日志`: must show retention parameters or irreversible impact before confirmation |

Operator-facing copy stays Chinese, direct, and outcome-first. Success toasts should prefer `已...` / `成功...`; failure toasts should include the backend-returned reason when available.

## Interaction Contract

- Keep the current page-scoped JavaScript ownership model. No component framework, client-side router, or DOM structure rewrite is allowed for Phase 3.
- Existing DOM ids/classes consumed by current scripts are compatibility surface. Preserve them unless the paired template and script change together in the same task.
- Loading states are fixed:
  - Tables/lists use skeleton rows or a single placeholder row inside the existing card/table shell.
  - Long-running actions switch button copy to `...中` or use the existing spinner helper.
  - `accounts_overview` card refresh keeps `aria-busy` and per-card loading lock.
- Empty states are fixed:
  - Use the shared `empty-state` pattern with icon + title + short next-step body in tables/cards.
  - `accounts_overview` card grid may keep its simpler `没有匹配的账号` sentence-only empty state.
- Success/failure feedback is fixed:
  - Mutations resolve with toast feedback.
  - Failures show a toast with explicit cause.
  - Page-level fetch failures render inline empty/error blocks, not browser alerts.
- Destructive actions are fixed:
  - Always gate delete/clear/cleanup/overwrite actions behind confirm UI.
  - Confirm copy must include target name or count.
  - If data loss is possible, the copy must say `此操作不可恢复` or the explicit backup behavior.
- Status communication is fixed:
  - Dense tables may use dot/badge shorthand only where already shipped.
  - Non-dense contexts (detail modal, service tables, stats, logs) must pair color with text or icon, not color alone.
- Scope guard is fixed:
  - Payment-related actions and Team links that already appear on management pages remain visible and unchanged.
  - Phase 3 must not redesign those entry points or move them to new pages.

## Page Contracts

### `/accounts`

- Keep the current page order: page header -> four stat cards -> one toolbar card -> one table card -> pagination -> detail/service-selection modals.
- Toolbar control order is fixed: filters/search on the left; `进入 Team 管理`, refresh, batch refresh, batch validate, batch check subscription, batch state, batch upload, batch delete, export on the right.
- Batch buttons must stay disabled until at least one account is selected, and their labels must append the selected count exactly as the current scripts do.
- Table columns remain dense and stable: checkbox, ID, email, password, email service, status, CPA, Sub2API, subscription, last refresh, actions.
- Detail view remains a modal, not a route transition.

### `/accounts-overview`

- Keep the current top summary stack: four stat cards -> four distribution cards -> quota/card-management block.
- The quota/card-management block keeps its current special treatment: gradient wrapper, icon-only actions, view toggle, plan filter, sort select, auto-refresh selector, add/import modals.
- Card removal is non-destructive to account records and must continue to say so in confirmation copy.
- `添加账号` and `一键导入账号` stay modal-based flows.

### `/settings`

- Keep the existing tab model and order: `代理设置` -> `访问控制` -> `上传` -> `Outlook配置` -> `注册配置` -> `验证码配置` -> `数据库`.
- Each tab remains a stacked-form/card layout inside the same page; no side navigation or split-pane redesign.
- Proxy management keeps the existing pattern: dynamic proxy form above proxy table, add/edit proxy in modal, destructive delete confirmed inline.
- Upload service management keeps one table per service type with add/edit modal flows and `测试连接` alongside `保存服务配置`.
- Database tools remain explicit admin actions with backup/import/cleanup grouped together.

### `/email-services`

- Keep the page order: stats -> collapsible Outlook batch import -> custom service table -> Outlook account table -> tempmail configuration cards -> add/edit modals.
- Outlook import panel stays collapsed by default and toggles between `展开` / `收起`.
- Custom services and Outlook services remain separate tables.
- Temporary email configuration remains card-in-card form layout, not modal-only.

### `/logs`

- Keep the vertical information flow: filters/control card -> stats card -> log stream card.
- Default control values remain: auto refresh `10 秒`, page size `100`, cleanup retention `30` days, cleanup max rows `50000`.
- Log stream stays a console-like mono panel with pagination below it.
- `立即清理` and `清空日志` remain distinct actions and distinct confirmations.

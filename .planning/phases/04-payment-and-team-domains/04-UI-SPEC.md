# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|

| Preset | not applicable; no `components.json`, `tailwind.config.*`, or `postcss.config.*` detected (source: codebase scan) |

---

## Phase Scope Lock

- Preserve the current visual language and interaction model; Phase 4 is backend-domain migration behind the existing UI, not a frontend rewrite.
- Keep current page routes, navbar placement, dark-mode toggle, card shell, tabs, tables, and modal patterns intact.
- Payment and team pages may receive only parity-preserving refinements that reduce ambiguity or improve operator feedback without changing page information architecture.
- Final cutover, router ownership sign-off, and production rollout messaging remain Phase 5 work and must not leak into page UX for this phase.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, label-to-control micro spacing, inline status dots |
| sm | 8px | Compact chip padding, small button gaps, table action gaps |
| md | 16px | Default form spacing, list-item padding, card-internal layout |
| lg | 24px | Card body padding, page section gaps, toolbar separation |
| xl | 32px | Page header separation, major stack spacing between top-level blocks |
| 2xl | 48px | Large subsection break when adding another payment/team block in this phase |
| 3xl | 64px | Full-page breathing room only; do not exceed without explicit product need |

Exceptions: 44px minimum height for team toolbar inputs and primary toolbar controls; 40px minimum height for dense bind-task row actions; existing desktop `team-panel` minimum height may stay at 640px as a structural layout exception (source: `static/css/style.css`, `templates/payment.html`).

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#f8fafc` | Page background, outer canvas, empty gutters, light-mode workspace |
| Secondary (30%) | `#ffffff` | Navbar, cards, modals, inputs, task rows, table surfaces |
| Accent (10%) | `#10a37f` | Active nav, primary CTA, selected tab/row outline, focus ring, team status emphasis, empty-state/callout highlight |
| Destructive | `#ef4444` | Delete, clear, remove, revoke, hard failure emphasis only |

Accent reserved for: current page nav item, payment primary generation/open actions, selected plan or selected Team emphasis, team overview eyebrow/chip highlights, focus rings, accepted/live-state callouts. Do not repaint all secondary buttons, tables, or passive labels with accent.

Local helper color contract: payment-specific purple `#8b5cf6` / `#7c3aed` may remain only for the selected payment plan outline, `批量识别` helper action, and `verifying`-style payment helpers. It is not a global brand color and must not replace the existing green primary system.

Semantic continuity: retain existing warning/info/success colors from `static/css/style.css` for status badges and toasts; do not collapse all state feedback into the accent color.

## Page Layout Contract

### Shared shell

- Keep the current sticky navbar, active-link highlight, page title, subtitle, and card-based main content shell.
- Keep shared light/dark theme behavior from `style.css`; Phase 4 must not introduce page-only theming rules or per-page palettes.
- Keep existing responsive breakpoints: desktop default, single-column collapse at `<=768px`, stacked full-width action layout at `<=480px`.
- Continue using tables for dense operational data; do not replace bind-task or task-center history with cards-only mobile redesigns.

### Payment page

| Area | Contract |
|------|----------|
| Top section | Keep `支付升级` header and `为账号生成 Plus 或 Team 订阅支付链接` subtitle. |
| In-page navigation | Keep exactly two tabs: `支付链接` and `绑卡任务`. Do not add sidebars, steppers, or wizard screens. |
| `支付链接` flow order | `选择套餐` -> `选择账号` -> `计费国家/货币` -> conditional Team options -> `绑卡资料` -> action row -> hidden result box. |
| Plan selection | Keep three plan cards `Plus`, `Business 试用`, `Team`; card selection remains visible and inline. |
| Bind-card data panel | Keep textarea-first parse flow, optional batch-preview table, and manual billing fields in one card. Do not move billing entry into modal or drawer. |
| Action row | Keep `生成支付链接` as primary, `生成并加入绑卡任务（半自动）` as secondary, with `创建任务后自动打开支付页` inline checkbox. |
| Result box | Keep result box hidden until a link exists. Its action set remains `直接打开支付页`, `复制链接`, `无痕打开浏览器`. |
| Task tab | Keep toolbar search + status filter + refresh action above the task table. Table column order stays `ID / 邮箱 / 套餐 / 状态 / Checkout / 来源 / 创建时间 / 操作`. |

### Team page

| Area | Contract |
|------|----------|
| Top section | Keep `自动进 Team` header and the current overview/detail/task-center positioning. |
| Header actions | Keep `发现母号` as primary and `批量同步` as secondary. |
| Filter toolbar | Keep `母号 ID`, `Team 状态`, `搜索 Team`, `刷新视图` in that order. |
| Desktop shell | Preserve the three-panel desktop workbench: overview list, detail/membership workbench, task center. Existing `1.2fr / 1fr / 1fr` balance remains the contract. |
| Overview panel | Keep metrics above the Team list. Each list item must show team name/status first, then owner and seats. |
| Detail panel | Keep six key facts, one operational callout, and the membership action area in the same panel. |
| Membership area | Keep tabs `活跃成员`, `邀请中`, `外部成员`; keep the invite availability chip and `批量邀请` trigger in the header. |
| Invite flow | Keep the invite modal pattern. Modes remain `邀请账号` and `邀请邮箱`; submit stays inside the modal footer. |
| Task center | Keep a live-status card above the historical task list; live card is the canonical place for accepted-task feedback. |

## Interaction Contract

- Keep button-local loading states. Payment and team actions may disable the initiating button or live area, but must not blank or full-screen lock the entire page.
- Success feedback stays dual-channel: toast for immediate acknowledgement plus a targeted inline update in the relevant panel, hint, or live card.
- Error feedback stays detail-first and context-local. Use toast plus the current panel's hint/status slot; do not redirect the operator away from the page.
- Search and filter changes refresh in place. Do not add explicit "Apply filters" steps for the current team page behavior.
- Destructive actions must require explicit confirmation before deletion, revoke, or remove operations.
- Accepted async flows must remain visible. For team discovery, batch sync, and invites, accepted responses immediately populate the task center and then transition through WebSocket updates without requiring manual polling.
- Payment result actions must preserve generated-link state until the operator changes upstream inputs. Do not auto-clear the result box after open/copy actions.

## Runtime Status Contract

### Payment bind-card task statuses

| Status value | Display label | UI treatment |
|--------------|---------------|--------------|
| `link_ready` | 待打开 | Informational badge; operator can open checkout immediately |
| `opened` | 已打开 | Warm state; operator has entered checkout context |
| `waiting_user_action` | 待用户完成 | Hold state; keep `我已完成支付` / `同步订阅` path visible |
| `paid_pending_sync` | 已支付待同步 | Positive-but-not-final; tell the operator to sync subscription |
| `verifying` | 验证中 | Transitional state; use helper purple only inside payment task UI |
| `completed` | 已完成 | Final success state |
| `failed` | 失败 | Final error state with visible reason |

### Team accepted-task live states

- Idle state copy: `暂无运行中的任务` and `当 discover / sync-batch 被 accepted 后，这里会监听 WebSocket 并在完成后刷新 Team 列表。`
- Submission failure copy: `提交失败: {detail}`.
- Socket failure copy: `监听失败` plus detail summary in the current-task description.
- Auto-discovery start copy: `正在自动发现母号 {id} 的 Team...`
- Completed task behavior: clear active task UUID, keep current Team selection, refresh teams/detail/tasks in place.
- Failed or cancelled task behavior: keep the current selection and last known data visible; do not reset the workbench to an empty state.

### Invite availability rules

- If `sync_status` is not `success`, disable invite actions and show `同步状态异常，请先完成一次成功同步再继续邀请。`
- If Team status is `full`, disable invite actions and show `当前 Team 已满，无法继续批量邀请。`
- Otherwise show the ready state and allow the invite modal to open without extra confirmation.

## Data-Visibility Contract

- Full card number and CVC stay transient in the current payment form flow and are never presented back in task tables; tables use masked values only.
- URLs, session diagnostics, and token-related content stay in monospace containers and remain copy-oriented, not visually emphasized as marketing content.
- Team overview/detail/task center must continue to expose backend truth rather than derived front-end-only counters whenever both are available.
- Backend field names and task status values remain unchanged; the frontend contract consumes the current JSON shape from `static/js/payment.js` and `static/js/auto_team.js` without renaming.
